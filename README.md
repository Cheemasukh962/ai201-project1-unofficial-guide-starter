# The Unofficial Guide — Project 1

## Domain

California community college transfer articulation agreements to UC campuses — specifically which ARC (American River College) courses satisfy UC Davis major preparation and General Education requirements.

This knowledge exists on ASSIST.org but is nearly impossible to navigate without prior experience. A student has to select the right school pair, choose the correct academic year, find the right major, and then manually read dense articulation tables. There is no way to simply ask "does my Calc 2 count?" and get a plain-English answer. This system makes that kind of question answerable in seconds, drawing directly from official articulation data.

## Chunking Strategy

**Chunk size:** Variable, averaging 119 characters (range: 60–381 chars). Not fixed — size is determined naturally by the structure of the data, not a preset character limit.

**Overlap:** None. Each chunk is an independent course equivalency pair. The calculus equivalency has no semantic relationship to the chemistry equivalency on the next line — overlap would only add noise.

**Why these choices fit your documents:**
ASSIST articulation data is structured as course equivalency pairs, not prose. Each pair looks like:

```
UC course:  MAT 021A - Calculus
CC course:  MATH 400 - Calculus I
```

This pair is atomic — splitting it would destroy its meaning. Chunking by fixed character count (e.g. 500 chars) would routinely merge multiple unrelated pairs or split a single pair across two chunks. Either way, retrieval breaks. One pair = one chunk is the only strategy that preserves the information unit. Documents were also deduplicated so that pairs appearing in multiple major files are only stored once.

**Preprocessing:** The scraper removes all HTML, navigation boilerplate, and API metadata before saving. Raw files contain only the header (`Major:`, `CC:`) and course pair lines. No further cleaning was needed.

**Final chunk count:** 1,332 deduplicated chunks across 100 source files.

## Embedding Model

**Model used:** `all-MiniLM-L6-v2` via `sentence-transformers`. Runs locally with no API key and no rate limits. Produces 384-dimensional vectors. Each of the 1,332 chunks is embedded once at build time and stored in ChromaDB on disk (`data/chroma/`).

**Production tradeoff reflection:**
For a real deployment serving thousands of students, I would evaluate `text-embedding-3-large` from OpenAI. It produces higher-quality embeddings for domain-specific text like course names and numbers, but costs money per embedding and requires an API call — not suitable for a one-time bulk embed on a student budget. I would also consider `multilingual-e5-large` if the system needed to serve Spanish-speaking students who might phrase queries in Spanish, since MiniLM is primarily English-optimized. The main tradeoff with larger models is latency: embedding a query at request time takes longer, which matters for a live UI. `all-MiniLM-L6-v2` is the right choice here because the corpus is small, cost and latency matter more than marginal accuracy gains, and the text is short structured English.

One observed limitation: cosine distances for this corpus tend to run between 0.40–0.85 rather than the sub-0.5 ideal. This is expected for very short text (60–150 chars) — the embedding vectors carry less semantic signal than paragraph-length chunks, so similarity scores are naturally lower. The correct answers are still retrieved; the distances just look higher than they would for longer text.

## Grounded Generation

**System prompt grounding instruction:**
```
You are a California community college transfer advisor.
Answer the student's question using ONLY the course articulation information provided below.
Do not use any outside knowledge — if the answer is not in the provided documents, say so.
Always cite which source document(s) your answer comes from.
```

The prompt explicitly forbids the model from using outside knowledge and requires a citation in every response. Temperature is set to 0.1 to reduce hallucination risk.

**How source attribution is surfaced in the response:**
Each retrieved chunk is passed to the LLM with its source filename prepended: `[Source: American_River_College_UC_Davis_Computer_Science_B.S..txt]`. The system prompt instructs the model to cite which document(s) the answer came from. Source filenames also appear in a separate "Retrieved from" field in the Gradio UI, surfaced programmatically regardless of what the LLM writes.

## Failure Case Analysis

**Question that failed:** "What main classes do I need at ARC to be transfer-ready for Computer Science at UC Davis?"

**What the system returned:** Only 2 courses (CISP 310 and CISP 400), both from unrelated major files (Biotechnology, Cognitive Science). The full CS prep list — which includes MATH 400, MATH 401, PHYS 350, and multiple ECS equivalents — was not returned.

**Root cause (tied to a specific pipeline stage):** Two compounding problems at the retrieval stage. First, the query is a broad, open-ended question — "transfer-ready for CS" — rather than a specific course lookup. The embedding of this query doesn't land close to any individual pair chunk because no single chunk covers "CS readiness." Second, the relevant chunks live in `Computer_Science_B.S..txt` and `Computer_Science_and_Engineering_B.S..txt`, but those files contain 15–26 pairs each. With top-k=5, only 5 chunks come back, and they come from whichever major files happened to embed closest to the broad query — not necessarily the CS-specific files.

**What you would change to fix it:** Two approaches. One: add a metadata pre-filter so that queries mentioning "computer science" route specifically to CS major files before running semantic search. Two: for multi-course "what do I need" questions, increase top-k to 15–20 and add a summarization step that groups results by major before passing them to the LLM.

## Spec Reflection

**One way the spec helped you during implementation:**
The chunking strategy section of planning.md was the most valuable part of the spec. Writing it out before touching code forced me to confront the core constraint: ASSIST data is structured pairs, not prose, and fixed-size chunking would destroy the information unit. Having this written down meant I could immediately rule out the standard LangChain text splitter and write a custom `_extract_pairs()` function instead. Without the spec, I might have started with a generic approach, seen bad retrieval, and spent hours debugging before realizing the root cause was at the chunking stage.

**One way your implementation diverged from the spec, and why:**
The spec listed institution IDs ARC=4 and UC Davis=79 based on early research. When the scraper ran, those IDs returned data for College of Marin and UC Berkeley — completely wrong. The actual IDs (ARC=27, UC Davis=89) were discovered by scraping the `/api/institutions` endpoint at runtime and searching for the correct institution names. The spec was wrong because the IDs were guessed rather than verified. This was caught before any data was saved, so no bad documents entered the corpus — but it meant the scraper needed a debugging pass before it could run successfully.

## AI Usage

**Instance 1 — scraper.py**

- *What I gave the AI:* The Documents section of planning.md (describing the ASSIST.org API as the data source, institution IDs, and the expected file format), plus the Mermaid architecture diagram showing the full pipeline.
- *What it produced:* A `requests`-based scraper that called the ASSIST API directly with the institution IDs from the spec.
- *What I changed or overrode:* The direct API calls returned HTTP 400 for every endpoint. The AI debugged through multiple header strategies (XSRF tokens, browser security headers, Content-Type) before discovering that the API blocked all non-browser requests. The final scraper used Playwright to run a headless Chrome browser, navigate the ASSIST UI, and intercept the API responses directly — a fundamentally different architecture from what was first generated. The parser also had to be rewritten after inspecting the actual JSON structure, which used double-encoded strings for the articulations field.

**Instance 2 — chunk.py and embed.py**

- *What I gave the AI:* The Chunking Strategy section from planning.md (specifying one-pair-per-chunk, no overlap, and the exact format of the raw .txt files), plus a sample document file showing the UC course / CC course line format.
- *What it produced:* `chunk.py` with a `_extract_pairs()` function that split on blank lines and extracted UC/CC line pairs, and `embed.py` that embedded chunks with all-MiniLM-L6-v2 and stored them in ChromaDB with metadata.
- *What I changed or overrode:* The initial version added a major-name prefix to every chunk text (e.g. `[Computer Science B.S.] UC course: ...`) to improve semantic context. After testing, this made distance scores worse rather than better — the extra words added noise to short embeddings. The prefix was removed and the major name was kept only in the metadata field, where it's used for filtering but not embedding. The deduplication logic was also added after noticing that the same course pair appeared in 10+ major files, flooding retrieval results with identical chunks from different sources.
